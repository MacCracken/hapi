# 2026-05-23 — P(-1) security audit

**hapi version**: 0.8.0 at scan time; repairs land in v0.8.x
toward v0.9.0.
**Auditor**: maintainer pass per CLAUDE.md Process P(-1) Step 5.
**Coverage**: every path argument's validation; every syscall's
return handling; every place a capability boundary is supposed
to bind a write.

## Scope

The four threat classes ADR 0005 + CLAUDE.md call out:

1. **Path traversal** — `..` segments slipping past the parser's
   validator or the cap-check.
2. **Symlink loops** — readlink chasing infinite chains during
   probe / link / adopt.
3. **TOCTOU** — time-of-check vs time-of-use races between a
   probe and the syscall that acts on it.
4. **Capability boundary** — does `--root` + `HAPI_ALLOWED_ROOTS`
   actually bound writes to permitted paths?

## Methodology

Read every `.cyr` source file in `src/` and `src/cmd/` with each
threat class as a lens. Demonstrated both confirmed findings end
to end against the v0.8.0 binary before writing repairs.

## Findings

### F-001 — `--root` `..` bypass [HIGH] — FIXED in v0.8.x

**What.** `cap_check_root_r` resolved the user-supplied `--root`
value to absolute via `fsl_to_absolute`, then byte-prefix-matched
against `$HOME`. `fsl_to_absolute` did *not* collapse `..`
segments, so a value like `/home/user/../etc/secret` passed the
matcher (first 10 bytes match `/home/user`; component boundary
splits on `/`) while actually denoting `/home/etc/secret` — a
path outside `$HOME`.

**Demonstrated.** Against v0.8.0 with `HOME=/tmp/probe/home/user`:

```sh
hapi link --root "/tmp/probe/home/user/../etc/secret" --dry-run pkg
# 0.8.0 emitted three "+ ...  (dry-run)" lines targeting
# /tmp/probe/home/user/../etc/secret/* — outside HOME.
```

**Impact.** Capability bypass — the load-bearing boundary per
CLAUDE.md *Key Principles*. Caller could write symlinks anywhere
the process had filesystem permission. Not a privilege-escalation
on its own (still bounded by Unix DAC), but defeats the explicit
"audit-the-flag-and-deny" contract ADR 0005 advertises.

**Fix.** New helper `fsl_lexical_normalize(path)` in
`src/fs_link.cyr` (Cyrius-side, no FS calls) that walks an
absolute path component-by-component, popping `..` segments and
dropping `.` segments. `cap_check_root_r` now calls it
immediately after `fsl_to_absolute`, both for the `--root` value
and for the `$HOME` value the matcher compares against. The
`_cap_in_allowlist` walker also lex-normalizes each entry so a
malformed `HAPI_ALLOWED_ROOTS=/etc/myproject/../escape` cannot
silently gate access to `/etc/myproject`.

**Verification.** Three new test groups land in v0.8.x:
- `audit-repair: lexical normalize collapses ..` (5 assertions)
- `audit-repair: cap check rejects .. that escapes $HOME` (3)
- `audit-repair: allowlist entries lex-normalized too` (2)

Re-running the bypass demo against the repaired binary:

```sh
hapi: --root path is not in HAPI_ALLOWED_ROOTS (and is not within $HOME)
```

### F-002 — `--root` symlink escape [MEDIUM] — DEFERRED

**What.** `cap_check_root_r` is *lexical only* even after F-001's
fix — it does not resolve symlinks. A path like
`/home/user/trap` where `trap` is a symlink to `/etc/secret`
passes the cap-check (the bytes match `/home/user/`), and the
write target later resolves through the symlink at kernel time.

**Demonstrated.** Against v0.8.0:

```sh
ln -s /tmp/probe/etc/secret /tmp/probe/home/user/trap
hapi link --root /tmp/probe/home/user/trap --dry-run pkg
# Writes "+ /tmp/probe/home/user/trap/.zshrc -> ..." — the
# actual on-disk effect would be at /tmp/probe/etc/secret/.zshrc.
```

**Impact.** Capability bypass under the *user's own filesystem
layout* — an attacker who can plant a symlink under `$HOME`
(typically via prior write access there) can redirect a future
`hapi link --root` to anywhere the process can write. Mediated
by Unix DAC; the attacker can't reach paths the process couldn't
already write.

**Defer rationale.** Real fix requires per-component symlink
resolution (open with `O_PATH | O_NOFOLLOW` at each segment, or
a hand-rolled `realpath` equivalent). That's enough code to want
its own design pass. ADR 0005 already names kavach as the
permanent owner of `cap_check_root_r`; kavach's capability
surface will do this properly. The v0.8.x lexical fix closes the
typo-class bypass (the F-001 case the dogfood user could trigger
without intent); the deliberate-attacker case waits for kavach.

**Tracked.** Filed at
[`docs/development/issues/2026-05-23-cap-check-symlink-escape.md`](../development/issues/2026-05-23-cap-check-symlink-escape.md)
for post-kavach repair. Severity Medium — accepted boundary at v1.0
per *kavach swap inside `src/cap.cyr`* in
[`state.md`](../development/state.md).

### F-003 — backup-copy TOCTOU symlink race [LOW] — FIXED in v0.8.x

**What.** `hapi_backup_copy(src, dst)` opened `src` with
`O_RDONLY` (no `O_NOFOLLOW`). The caller (cmd_link / cmd_adopt)
already probed `src` to confirm it's a regular file, but an
attacker racing a `mv symlink → src` between probe and copy
could redirect the backup to follow a symlink to anywhere
readable by the process. The backup file would contain
attacker-chosen content rather than the user's original bytes.

**Impact.** The user-visible damage is "backup contains the
wrong content" — a snapshot of the wrong file. Severity Low: the
attacker has to already be running on the system to win the race,
and the audit `backup_path` field still records where the bytes
went. But the documented contract is "snapshot of the original
file's bytes," so defending the contract is cheap.

**Fix.** Source open switched to
`file_open(src, O_RDONLY | O_NOFOLLOW, 0)`. O_NOFOLLOW = 0x20000
on Linux x86_64. ELOOP on the open path now produces a clean
`backup-to copy failed` exit (1) instead of a silent
content-substitution.

**Verification.** New test group `audit-repair: backup_copy
refuses symlink source (TOCTOU defense)` (2 assertions) plants
a symlink + the regular file behind it and confirms the symlink
copy is refused while the regular-file copy succeeds.

### F-004 — Audit-trail trust model [INFO] — ACCEPTED

**Threat.** An attacker who can write
`$XDG_STATE_HOME/hapi/audit.jsonl` could forge entries — e.g. an
`op:unlink` for a target that doesn't yet exist, so the next
`hapi rollback` would emit a fresh `op:link` writing a
symlink at the attacker's chosen path.

**Mitigation.** ADR 0002 already calls this out and rejects
per-entry signing as out of scope: *"if an attacker can write to
`~/.local/state/hapi/`, they can rewrite hapi's binary too."*
The threat model — local user trusts their own state directory
— stays the same.

**Bounds rollback could exercise.** rollback's reversal helpers
(`_hrb_reverse_link`, `_hrb_reverse_unlink`, `_hrb_reverse_adopt`)
each precondition-check the on-disk state against the audit
entry. A forged `op:unlink` targeting `/etc/passwd` is harmless
(`/etc/passwd` exists, so the reverse-unlink path skips). A
forged `op:unlink` targeting a path that doesn't exist would
attempt to create a symlink there — bounded by Unix DAC on the
parent directory.

**Defer rationale.** The accepted boundary is "trust the audit
trail" — same as trusting the hapi binary itself. No code change.

### F-005 — TOCTOU between probe and act [INFO] — ACCEPTED

**Threat.** Every mutating verb does a `link_probe` first and a
syscall (sys_unlink / sys_symlink / sys_rename) second. Between
those two calls another process can modify the target.

**Mitigation.** Single-process semantics. The user is responsible
for not concurrently modifying their dotfiles while running hapi.
The race window is also tiny: probe → maybe-backup → unlink →
symlink — microseconds in the typical case.

**Defer rationale.** Defending this would require flock-on-target
or "rename into place" tricks that don't compose with how the
filesystem represents user-facing paths. Accepted boundary.

## Sign-off

The repaired v0.8.x binary no longer exhibits the demonstrated
F-001 bypass. F-003's defense is asserted by the new test group.
F-002 is the documented kavach-time follow-up; no consumer
relies on cap_check_root_r resolving symlinks today.

This pass closes CLAUDE.md Process P(-1) Step 5 for v1.0 — re-run
on any new code that touches a path argument, a syscall, or the
capability surface.
