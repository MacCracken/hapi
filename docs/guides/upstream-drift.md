# Upstream drift — adopting a config whose stock template keeps changing

Some upstream tools (hyprland is the canonical case; sway, river,
fish to a lesser extent) **rewrite their stock-config template
every few releases**. When you adopted the file last cycle, you
got the *then-current* upstream syntax plus your customizations.
When you reinstall or reconfigure on the new release, the on-disk
fresh-install copy is the *current* upstream syntax — sometimes
with new keywords, new block structure, new defaults you want to
know exist.

`hapi link --force <pkg>` from the older repo file would silently
overwrite the newer on-disk file, rolling back the upstream's
evolution without telling you. The right move is an **audit +
merge** pass before the `--force`. This guide codifies that ritual.

## When this applies

Reach for this workflow when **all three** are true:

1. The upstream tool ships a stock-config template (usually under
   `~/.config/<tool>/`).
2. Diffing your repo copy against the on-disk copy shows
   *structural* differences (new keywords, reformatted blocks,
   new default options) — not just content differences (your
   keybinds vs. their stock keybinds).
3. The upstream copy carries an `autogenerated = 1` header or
   similar "this template will be rewritten on upgrade" marker.

Known-drifting upstreams as of 2026-05: **hyprland** (every few
releases), **sway** + **river** (occasional), **fish** (rare),
**kitty** (very rare). Tools like vim, tmux, git, kitty's older
configs, and most shell rcs evolve additively and don't need this
ritual.

## The ritual

```sh
# 1. Snapshot the on-disk fresh template before any --force.
cp ~/.config/hypr/hyprland.conf ~/.config/hypr/hyprland.conf.pre-hapi.bak

# 2. Diff repo (older stock + your mods) vs on-disk (newer stock).
diff -u ~/.config/hypr/hyprland.conf ~/dotfiles/hyprland/.config/hypr/hyprland.conf | less

# 3. Read end-to-end. Sort each diff hunk into one of two piles:
#    - Upstream syntax shift  → adopt from the on-disk side.
#    - Your customization     → adopt from the repo side.

# 4. Write the merged file at the repo location. Base = newer
#    upstream stock; layered on top = your customizations.
$EDITOR ~/dotfiles/hyprland/.config/hypr/hyprland.conf

# 5. Add a tracking-note header to the merged file (see below).

# 6. Validate the manifest still parses cleanly.
hapi check --strict ~/dotfiles/hyprland

# 7. Preview the apply. (link --dry-run is the right pre-flight;
#    see status.md for why `status &&` doesn't compose here.)
hapi link --dry-run ~/dotfiles/hyprland

# 8. Apply.
hapi link --force ~/dotfiles/hyprland

# 9. Verify clean state.
hapi status ~/dotfiles/hyprland

# 10. After the new shell / login confirms the new file works,
#     delete the .pre-hapi.bak snapshot.
rm ~/.config/hypr/hyprland.conf.pre-hapi.bak
```

## Identifying upstream syntax shifts

Read the diff top-down. Syntax shifts cluster around these
markers — when you see them on the on-disk side and not the
repo side, the on-disk side is newer:

- **Wiki / docs URLs** — `wiki.hyprland.org` → `wiki.hypr.land`.
  Cosmetic but a reliable freshness signal.
- **Header markers** — `autogenerated = 1 # remove this line to
  remove the warning`. Upstream adds this when it wants users
  to know the file gets rewritten.
- **Block-syntax migrations** — single-line directives moving
  to block form (`windowrule = key val, key val` →
  `windowrule { name = ... }`).
- **New keyword names** — same concept, renamed
  (`gesture = ...` → `gestures = ...` with a different template).
- **New default options** — fields that didn't exist last
  release (e.g. `rounding_power = 2` under `decoration`).
- **New commented sections** — `# Permissions /
  # Smart-gaps / # ...` blocks present as commented templates
  for reference.

These are upstream concerns. Adopt them.

## Identifying your customizations

User-customization differences live inside otherwise-stable
structure:

- Keybind targets (`bind = ..., exec, foo` where `foo` is yours).
- Monitor blocks (`monitor=DP-1, 3440x1440@165, ...`).
- `exec-once = ...` lines for your session setup.
- Brightness / volume increment values.
- Custom `windowrule` entries for apps the stock template doesn't
  know about.

These are your concerns. Adopt them.

## The tracking-note header

After the merge, bake a header comment into the merged file so
the *next* drift cycle has a starting point. The convention:

```
# ----------------------------------------------------------------
# hapi: drift-audit 2026-05-23 — absorbed upstream changes:
#   - wiki URL migration (hyprland.org → hypr.land)
#   - windowrule block-syntax migration
#   - gestures keyword rename + template
#   - decoration.rounding_power default
#   - bindel volume cap (-l 1)
#   - commented Permissions / Smart-gaps reference blocks
# Re-audit when on-disk fresh template no longer matches this
# file's syntax shape (next hyprland minor / major bump).
# ----------------------------------------------------------------
```

The dated note tells future-you which release cycle you last
audited against. When the next stock rewrite drops, diffing the
upstream-current file against this annotated copy makes the new
syntax shifts pop out immediately.

## Why hapi doesn't automate this

The merge is *semantic*: it needs to know which side has the
"newer syntax" and which has "the user's intent." A 3-way merge
driver could scaffold the diff, but it can't decide which side
each hunk should come from — that's the audit step.

A speculative `hapi merge <pkg>` verb sits in the roadmap's
*Deferred (post-v1.0)* list under
[`issues/archived/...upstream-drift-pattern.md`](../development/issues/archived/2026-05-20-upstream-stock-template-drift-pattern.md)
Tier-3. It needs a clear data model for "what is a stock-template
version" before earning its place — for v1.0, the manual ritual
plus the tracking-note header is the surface.
