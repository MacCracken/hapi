# Upstream stock-template drift (hyprland-style) needs a documented audit workflow

**Discovered:** 2026-05-20 during M7 dotfile dogfooding (hyprland package adoption)
**Severity:** Low — process / guide-level surface, not a hapi runtime bug
**Affects:** any consumer adopting a package where upstream rewrites its stock-config template every few releases

## Summary

Some upstream tools (hyprland is the canonical case; sway, river, fish to a lesser extent) **rewrite their stock-config template every few releases**. The user's dotfile-repo copy of `hyprland.conf` may diverge from the on-disk fresh-install copy not just in *content* (the user's keybinds, monitors, etc.) but in *syntax / structure*:

- New keyword names (`gesture` → `gestures` and back)
- Reformatted block structure (`windowrule = key val, key val` → `windowrule { name = ...\n match:class = ...\n  suppress_event = ... }`)
- New required preambles (`autogenerated = 1 # remove this line to remove the warning`)
- Wiki URL changes (`wiki.hyprland.org` → `wiki.hypr.land`)
- New default options (`rounding_power = 2` in the decoration block)

A simple `hapi link --force` from the user's existing repo file **overwrites the on-disk newer-syntax stock copy with the older-syntax user copy**, silently rolling back the upstream's evolution. That works at runtime if hyprland still parses the old syntax, but it loses every new option the user might want to know exists.

The M7 dogfood session worked around this by doing a manual audit + merge: diff the two files, identify which side has the *newer syntax* (upstream), which side has the *user customizations* (repo), and produce a merged file with both. The audit was a real cost (took 10+ minutes of careful reading for one config) and is repeated every time the user reinstalls hyprland.

This is a *process* gap, not a hapi-code gap. But hapi is the tool the user reaches for when adopting these files, so the workflow probably wants documentation in the hapi guide tree — and possibly automation later if the pattern proves universal.

## Reproduction

Concrete shape from the M7 session:

| Aspect | Repo (older stock + user mods) | On-disk (fresh install, newer stock) |
|---|---|---|
| Wiki URLs | `wiki.hyprland.org` | `wiki.hypr.land` |
| Header | clean | `autogenerated = 1` preserved warning |
| `windowrule` | mix of short + block | all block |
| `gesture` | working binding `gesture = 3, horizontal, workspace` | metavar template `gestures = fingers, direction, action, options` |
| `decoration` | no `rounding_power` | `rounding_power = 2` |
| `bindel volume` | no limit-cap | `-l 1` cap-at-100% flag |
| Permissions section | absent | commented template present (reference) |
| Smart-gaps section | absent | commented template present |

The naive `hapi link --force` would have overwritten the entire newer-syntax on-disk file with the older-syntax repo file, losing 7+ upstream improvements.

## Root cause

Not a hapi bug. The friction is in the *dotfile-management workflow* when the upstream tool aggressively rewrites its own config schema. Other tools (vim, tmux, git, kitty) don't do this — they evolve more conservatively or via additive options.

Hapi's contribution to the fix is **process + tooling**, not runtime semantics:

1. **Documented merge workflow** — a guide showing the audit pattern (diff → identify user-mods → identify upstream-syntax-changes → merge with both → `link --force` the merged file).
2. **A tracking-note comment** baked into the merged config explicitly warning future readers that upstream rewrites this template; when the next stock-template drift happens, re-audit via the same workflow.
3. **(Speculative) `hapi merge <pkg>`** — a future verb that scaffolds the merge by reading the repo file, the on-disk file, and a stock-template path, producing a 3-way merge proposal. Probably too tool-specific for hapi's v1.0 surface; might earn a sibling tool (`hapi-merge` or similar) if the pattern proves load-bearing.

## Proposed fix

### Tier 1 (now): new guide `docs/guides/upstream-drift.md`

The dogfood session captured the workflow. Codify it as a hapi guide:

- *When this applies* — list of known-drifting upstreams (hyprland + sibling family, sway, river, fish; basically anything that ships an `autogenerated = 1` warning).
- *The audit step* — `diff -u <on-disk> <repo>`, read both ends.
- *Identifying upstream syntax shifts* — file format markers (wiki URLs, header comments, block-syntax migrations, new default options).
- *Identifying user customizations* — keybinds, monitors, exec-once lines, brightness/volume increments.
- *Writing the merged file* — repo file becomes the merged version; backup the on-disk fresh template; `link --force` to apply.
- *Tracking-note comment* — bake a header into the merged file noting the audit date + the syntax shift class so the next drift cycle has a starting point.

### Tier 2 (post-v1.0): architecture note `docs/architecture/NNN-upstream-drift-pattern.md`

If the pattern becomes load-bearing across multiple packages, formalize it as an architecture note — what invariants hapi holds when adopting drifting-upstream files, what's deliberately NOT automated, what consumers of hapi-managed packages can rely on.

### Tier 3 (speculative): `hapi merge <pkg>` verb

A 3-way merge driver that scaffolds the audit. Probably outside v1.0 scope; would need a clear data model for "what is a stock-template version" (timestamp? hash? hapi-tracked separately?) before earning its place.

## Consumer-side workaround

What the M7 dogfood session did, captured here for future use:

1. `cp <on-disk-file> <on-disk-file>.pre-hapi.bak` — safety net.
2. `diff -u <on-disk-file> <repo-file> | less` — read end-to-end.
3. Identify *upstream syntax shifts* (read top-down, syntax/structure differences).
4. Identify *user customizations* (content differences inside otherwise-stable structure).
5. Write a merged file at the repo location — base = newer upstream stock, layered = user customizations.
6. Add a tracking-note header comment naming the audit date + the syntax shifts absorbed.
7. `hapi check --strict <pkg>` against the new merged file (catches typos).
8. `hapi link --dry-run <pkg>` to preview.
9. `hapi link --force <pkg>` to apply.
10. `hapi status <pkg>` to verify clean state.

The 10-step ritual is too much friction to do without a guide doc; codify in Tier 1 above.
